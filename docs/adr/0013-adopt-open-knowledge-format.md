# Adopt Open Knowledge Format (OKF)

## Context
In an agentic architecture relying on Markdown for Context Scoping (like the Interpretable Context Methodology), AI agents continuously generate and consume documents. However, pure Markdown files are opaque to deterministic orchestrators (like Turnlock). This creates the "context-assembly problem": orchestrators cannot programmatically validate the document's type, its inputs, or its semantic intent without invoking an expensive and unreliable LLM parse. 

## Decision
Adopt the **Open Knowledge Format (OKF)** standard (v1.0) universally across the agentic framework. 
Every agent-facing document must include a strictly typed YAML frontmatter that defines its schema. The core types (`kind`) to be adopted are:
1. `AgentWorkflowStep` (for routing and input/output contracts)
2. `AgentSkill` (for global capabilities)
3. `KnowledgeAsset` (for passive conventions and rules)
4. `RuntimeArtifact` (for data generated by agents during a session)

## Alternatives Considered
- **Pure JSON for metadata / Markdown for content (Separate files)**: Keeping the metadata in a separate `.json` file alongside the `.md` file was rejected because it fragments the state and makes manual "Glass Box" editing error-prone for human developers.
- **Unstructured YAML Frontmatter**: Allowing arbitrary YAML tags without a strict schema was rejected because it prevents deterministic orchestrators from enforcing data contracts.

## Consequences

### Pros
- **Bridging Determinism and NLP**: Allows strict state machines (Turnlock) to instantly parse, validate, and route Markdown documents programmatically using the YAML metadata, while the LLM consumes the human-readable Markdown body.
- **Contract Enforcement**: By defining the `format` in a `RuntimeArtifact`, the orchestrator can mechanically guarantee that the output of Step 1 exactly matches the required input of Step 2 before launching the agent.
- **Self-Describing System**: The entire workspace becomes a highly indexed, semantic graph that both humans and agents can navigate effortlessly.

### Cons
- Adds a strict boilerplate requirement to every document, forcing developers and agents to maintain accurate YAML schemas manually.
